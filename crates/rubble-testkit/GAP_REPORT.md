# Rubble physics engine — gap report

Generated by the `rubble-testkit` detection harness. This is the deliverable: an
explicit, categorized account of what the engine currently gets right, what it
does **not**, and how confident we are in each conclusion. The goal of this pass
was to *detect* bugs and map gaps — not to fix them.

Run the live version yourself:

```bash
# software Vulkan (CI-equivalent)
sudo apt-get install -y mesa-vulkan-drivers
export WGPU_BACKEND=vulkan
export VK_ICD_FILENAMES=$(find /usr/share/vulkan/icd.d -name 'lvp_icd*' | head -1)

# native lane — prints the full per-scenario report (green on catalogued gaps)
cargo test -p rubble3d --test harness_tests -- --nocapture

# surface every catalogued gap as a failure (the known-failures lane)
RUBBLE_RUN_KNOWN_FAILURES=1 cargo test -p rubble3d --test harness_tests

# fault-detection matrix — proves the suite catches injected bugs
cargo test -p rubble3d --test fault_detection_tests -- --ignored --nocapture
```

## How to read a failure

Every check is **derived, not fitted**: each tolerance comes from f32 round-off
accumulation or a config-defined physical limit (`penetration_slop`,
`contact_offset`), never from "loosen until it passes". A failure prints the
exact `tick`, `body`, `measured` and `bound`, and a failing run carries its full
per-tick trajectory. So a failure is unambiguous: it is a real discrepancy from a
documented expectation, not a hand-picked threshold that drifted.

The two-tier convention: **tight derived** bounds (quaternion norm, non-
penetration, energy non-increase, momentum, ballistics) are real-bug detectors;
**loose sanity** bounds (explosion/teleport guards) are catastrophe detectors
that should never trip in correct behavior.

## Engine gaps DETECTED (catalogued in `gaps.rs`)

### 1. Systematic angular-velocity drift under zero torque — `AngularIntegration`
`zero_g_sphere_constant_spin`: ω drifts **~3.3e-3 over 180 zero-torque steps**,
**identically for a sphere and a box**. A sphere is isotropic and cannot precess,
so the shape-independence proves this is a **systematic quaternion integration /
extraction bias, not gyroscopic free precession**. It exceeds the random-walk f32
floor (~2e-4) by ~17×.

This refines the existing `zero_gravity_shapes_preserve_velocity_and_spin_3d`
known-failure, which attributes the drift to "3D gyroscopic integration" and
incorrectly expects *boxes* to keep constant ω — physically a box's ω **should**
evolve (tennis-racket theorem); only its energy and angular momentum are
conserved. The real bug is the shape-independent numerical drift.

### 2. Floor penetration under combined normal+tangential load — `Solver`
`static_friction_holds`: a box resting on the floor sinks **~0.15 m below the
surface from the first tick** under tilted gravity (a normal + large tangential
load). The straight-gravity equivalent (`resting_box_settles`) does **not**
penetrate, isolating this to the contact solver under-resolving the normal
constraint when a strong tangential load is present. (Magnitude should be
confirmed independently with the Parry depth oracle — see "Not yet covered".)

## Suite blind spots (from the fault-detection matrix)

The matrix injects known bugs and checks the suite catches them:

| Injected fault | Caught by |
|---|---|
| `NegateGravity` | free_fall, projectile, drop_bounce, resting_box, stack |
| `ZeroGravity` | free_fall, projectile |
| `ScaleGravity(1.5)` | free_fall, projectile, drop_bounce, resting_box |
| `DropSolverIterations` (→1) | two_body, drop_bounce, stack |
| `ZeroFriction` | **NOT DETECTED** |

4/5 faults are caught by multiple scenarios — the derived tolerances genuinely
have teeth. **`ZeroFriction` is a blind spot**: the only friction scenario
(`static_friction_holds`) is quarantined for gap #2, and the resting/stack
scenarios carry no tangential load, so zeroing friction changes nothing the
*passing* suite observes. **Friction is under-tested** until gap #2 is fixed or a
gentle-incline scenario that passes-with-friction is added.

## Methodology findings (the tests were wrong before the engine was)

Two of the first failures were errors in the *tests*, not the engine — exactly
the "how do we know the tests test the right thing?" risk. Both were fixed with
derivations, not looser numbers:

- **Velocity tolerance was too tight.** Velocity is *extracted* by finite-
  differencing f32 positions (`v=(xₙ−xₙ₋₁)/dt`), which amplifies position round-
  off by ~1/dt and scales with position magnitude. The fix derives the bound as
  `pos_scale · ε_f32 / dt`, not a flat number.
- **"Constant spin" is invalid for anisotropic bodies.** Asserting constant ω on
  a tumbling box is physically wrong (free precession). The oracle now applies
  constant-ω only to isotropic spheres and uses energy conservation for the box.
  This is also why gap #1 is credible: the sphere isolates the *numerical* drift
  from the *physical* precession.

The fault matrix is what keeps the (now-correct) tolerances honest going forward:
if a tolerance is ever loosened too far, an injected fault stops being caught and
the matrix turns red.

## Not yet covered (next steps)

- **Independent geometry oracle (Parry) across all shape pairs.** A `parry3d`
  contact oracle exists (`crates/rubble3d/tests/support/parry_oracle.rs`) but is
  barely used; expanding it across sphere/box/capsule/convex/plane at random
  poses would give a tolerance-defensible narrowphase check and independently
  confirm gap #2's penetration depth.
- **Browser lane (the primary user surface).** A wasm `run_scenario_*` entry
  point running the same scenarios + checks inside wasm (one `await`, not per-
  tick `page.evaluate`) under Chrome+SwiftShader, so native-vs-browser
  divergences (toolchain / binding / async / device-limit) surface as a
  `BrowserOnly` gap category.
- **A passing friction scenario** to close the `ZeroFriction` blind spot.
- **2D coverage** — the harness is structured to extend to `rubble2d`.
- **Convergence testing** for rotational accuracy (does gap #1 shrink at the
  integrator's order as dt→0?), which would distinguish "acceptable order-of-
  accuracy" from "biased integrator".
